# 1. Overview

Cryptographic code fails silently. A field arithmetic bug rarely produces a
crash. Instead it produces a wrong but plausible-looking result that passes
casual review and only surfaces as a soundness failure much later, often in
production. Standard unit tests with hand-picked inputs are not sufficient
to catch this class of bug.

This library uses two complementary testing strategies beyond ordinary unit
tests:

* **Property-based testing** with `proptest`, which generates randomized
  inputs and checks that algebraic invariants hold across the input space.
* **Differential testing**, which runs the same computation through this
  library and an independent reference implementation, then asserts the
  outputs are identical.

This document explains how to run both suites, how to write new tests in
each style, and what is expected of any PR that touches arithmetic code.

# 2. Prerequisites

Before running either suite, ensure your environment matches the setup
described in the main `README.md`:

~~~
rustup target add wasm32-unknown-unknown
cargo install --locked soroban-cli
~~~

The differential testing suite additionally requires the reference crate
used for cross-checking. This is declared as a dev-dependency and is
installed automatically when you run `cargo test`, but requires network
access on first build:

~~~
cargo fetch
~~~

# 3. Property-Based Testing with `proptest`

## 3.1 What Property Tests Check

A property test does not check a single fixed input and output. It states
a mathematical property that must hold for any valid input, then generates
hundreds of random inputs and verifies the property on each one.

For field arithmetic, typical properties include:

* **Additive identity:** $a + 0 = a$ for any field element $a$
* **Additive inverse:** $a + (-a) = 0$
* **Commutativity:** $a + b = b + a$ and $a \cdot b = b \cdot a$
* **Associativity:** $(a + b) + c = a + (b + c)$
* **Distributivity:** $a \cdot (b + c) = a \cdot b + a \cdot c$
* **Round-trip conversion:** `from_mont(to_mont(a)) == a`
* **Inversion correctness:** $a \cdot a^{-1} = 1$ for any nonzero $a$

## 3.2 Running the Existing Suite

~~~
cargo test --package zk-core --lib proptest
~~~

This runs every function annotated with `proptest!` across the `zk-core`
crate. By default each property is checked against 256 randomly generated
cases. Failures are automatically minimised by `proptest` to the smallest
input that reproduces the failure, called **shrinking**.

To run a single property test module, for example the $\mathbb{F}_{q^2}$
arithmetic properties:

~~~
cargo test --package zk-core --lib fq2::proptests
~~~

## 3.3 Increasing Case Count for Deeper Search

The default case count is intentionally low for fast local iteration.
Before submitting a PR that touches arithmetic, run with a higher case
count to search a wider input space:

~~~
PROPTEST_CASES=10000 cargo test --package zk-core --lib proptest
~~~

## 3.4 Writing a New Property Test

Property tests live alongside the code they test, in a `proptests` module
gated behind `#[cfg(test)]`. The structure follows this pattern:

~~~
#[cfg(test)]
mod proptests {
    use proptest::prelude::*;
    use super::*;

    proptest! {
        #[test]
        fn add_is_commutative(a in any_fq2(), b in any_fq2()) {
            prop_assert_eq!(fq2_add(a, b), fq2_add(b, a));
        }

        #[test]
        fn mul_distributes_over_add(a in any_fq2(), b in any_fq2(), c in any_fq2()) {
            let lhs = fq2_mul(a, fq2_add(b, c));
            let rhs = fq2_add(fq2_mul(a, b), fq2_mul(a, c));
            prop_assert_eq!(lhs, rhs);
        }

        #[test]
        fn inversion_round_trips(a in any_nonzero_fq2()) {
            let inv = fq2_inv(a);
            prop_assert_eq!(fq2_mul(a, inv), fq2_one());
        }
    }
}
~~~

## 3.5 Custom Strategies

Random field elements are not generated by drawing arbitrary bytes. They
must be generated within the valid range $[0, p)$ to avoid testing
ill-formed inputs that would never occur in practice. Use the provided
strategy helpers rather than writing new ones:

~~~
fn any_fq2() -> impl Strategy<Value = Fq2> {
    (any_fq(), any_fq()).prop_map(|(a0, a1)| Fq2::new(a0, a1))
}

fn any_nonzero_fq2() -> impl Strategy<Value = Fq2> {
    any_fq2().prop_filter("nonzero", |x| !x.is_zero())
}
~~~

If you are testing a new field or curve type, add the corresponding
strategy to `zk-core/src/testutils/strategies.rs` rather than duplicating
generator logic in individual test modules.

## 3.6 What a Property Test Cannot Catch

Property tests verify internal consistency: if your implementation has a
sign error that is wrong but self-consistent across addition, subtraction,
and multiplication, the algebraic properties can still hold. Property
testing alone is not sufficient for cryptographic correctness. This is
what differential testing is for.

# 4. Differential Testing

## 4.1 What Differential Testing Checks

Differential testing runs the same input through two independent
implementations and asserts the outputs match. For this library, the
reference implementation is a well-established external crate with the
same field and curve parameters. Any divergence between this library's
output and the reference output indicates a bug in one of the two
implementations, almost always this library's.

## 4.2 Running the Differential Suite

~~~
cargo test --package zk-core --lib differential --features differential-testing
~~~

The `differential-testing` feature flag gates the reference crate
dependency so it is never pulled into the production `no_std` build. This
flag must never be enabled outside test builds.

## 4.3 Suite Coverage

The current differential suite covers:

| Module | Reference comparison |
|--------|----------------------|
| $\mathbb{F}_q$, $\mathbb{F}_r$ arithmetic | Field operations cross-checked against the reference crate's prime field implementation |
| $\mathbb{F}_{q^2}$, $\mathbb{F}_{q^6}$, $\mathbb{F}_{q^{12}}$ arithmetic | Extension field operations cross-checked component-wise |
| $G_1$, $G_2$ point operations | Addition, doubling, and scalar multiplication cross-checked in both Jacobian and affine form |
| BN254 pairing | Single pairing evaluation cross-checked against the reference Miller loop and final exponentiation |
| NTT | Forward and inverse transform cross-checked against direct polynomial evaluation for small $n$ |

## 4.4 Writing a New Differential Test

~~~
#[cfg(all(test, feature = "differential-testing"))]
mod differential {
    use super::*;
    use reference_crate::Fq2 as RefFq2;
    use proptest::prelude::*;

    fn to_reference(a: Fq2) -> RefFq2 {
        RefFq2::new(a.c0.into(), a.c1.into())
    }

    proptest! {
        #[test]
        fn mul_matches_reference(a in any_fq2(), b in any_fq2()) {
            let ours = fq2_mul(a, b);
            let theirs = to_reference(a) * to_reference(b);
            prop_assert_eq!(to_reference(ours), theirs);
        }
    }
}
~~~

The conversion functions `to_reference` and `from_reference` must be
defined once per type and reused. Duplicating ad hoc conversion logic
across test modules has historically been a source of test bugs.

## 4.5 When Differential Testing Is Required

Any PR that adds or modifies one of the following must include a
differential test before it can be merged:

* Field arithmetic (addition, multiplication, squaring, inversion)
* Curve point arithmetic (addition, doubling, scalar multiplication)
* Pairing computation (Miller loop, final exponentiation, sparse multiplication)
* NTT or polynomial evaluation
* Any Montgomery reduction or conversion routine

PRs that are pure documentation, refactoring with no behavioural change,
or changes to non-arithmetic code (such as Soroban host bindings) are
exempt.

# 5. Fuzzing for Edge Cases

While `proptest` generates random inputs uniformly, certain bugs only
appear at the boundaries of the valid input range: values near zero, near
the field modulus, or at points where intermediate computations might
overflow before reduction.

## 5.1 Boundary Strategies

When writing a property test for a new operation, include explicit
boundary cases alongside the random strategy:

~~~
proptest! {
    #[test]
    fn add_handles_near_modulus(a in any_fq2()) {
        let near_p = Fq2::new(Fq::from(P_MINUS_ONE), Fq::from(P_MINUS_ONE));
        let result = fq2_add(a, near_p);
        prop_assert!(result.is_canonical());
    }
}
~~~

`is_canonical()` checks that the result lies in $[0, p)$ for both
components. This catches reduction bugs that only manifest when an
addition would otherwise overflow past the modulus.

## 5.2 Regression Corpus

When `proptest` shrinks a failing case to a minimal counterexample, persist
it as a regression test rather than relying on `proptest`'s seed file alone:

~~~
#[test]
fn regression_fq2_mul_overflow_case() {
    // Discovered by proptest 2026-XX-XX, shrunk from a 10000-case run.
    // See issue #XXX for context.
    let a = Fq2::new(Fq::from_hex("..."), Fq::from_hex("..."));
    let b = Fq2::new(Fq::from_hex("..."), Fq::from_hex("..."));
    let result = fq2_mul(a, b);
    assert!(result.is_canonical());
}
~~~

This ensures the specific failing case remains covered even if the random
seed used by `proptest` changes between CI runs.

# 6. Running the Full Suite Before a PR

Before opening a PR, run the complete sequence used by CI:

~~~
make clippy
make fmt
cargo test --workspace
PROPTEST_CASES=10000 cargo test --package zk-core --lib proptest
cargo test --package zk-core --lib differential --features differential-testing
make build-wasm
~~~

All of these must pass with zero failures and zero warnings before the
PR checklist can be honestly checked off.

# 7. Common Failure Patterns

## 7.1 Property Test Fails Only at High Case Counts

If a property test passes at the default case count but fails at
`PROPTEST_CASES=10000`, this almost always indicates a bug at a rare
boundary condition. Do not increase the default case count to hide this.
Find and fix the boundary case, then add it as a regression test per
Section 5.2.

## 7.2 Differential Test Fails on Specific Inputs Only

If the differential suite fails on specific inputs but property tests
pass, the bug is consistent with itself but disagrees with the reference
semantics. This is the most dangerous class of bug, since it can pass
extensive internal testing while still being cryptographically wrong.
Trace the divergence to the exact operation where the reference and this
library's intermediate values first differ.

## 7.3 Differential Test Build Failures

If `cargo test --features differential-testing` fails to compile, check
that the reference crate version pinned in `Cargo.toml` is still
compatible with your Rust toolchain version. The reference crate is a
dev-dependency only and version mismatches here do not affect the
production build.

---

# 8. Summary

Every contribution that touches arithmetic code must pass both testing
layers before merge:

| Layer | Purpose | Command |
|-------|---------|---------|
| `proptest` | Verifies algebraic invariants across randomized inputs | `cargo test --lib proptest` |
| Differential testing | Verifies outputs match an independent reference implementation | `cargo test --lib differential --features differential-testing` |
| Regression corpus | Locks in previously discovered failing cases | Part of the standard `cargo test` run |

Property tests catch internal inconsistency. Differential tests catch
correctness relative to an external source of truth. Neither is a
substitute for the other, and both are required for any PR modifying
field, curve, pairing, or polynomial arithmetic.
